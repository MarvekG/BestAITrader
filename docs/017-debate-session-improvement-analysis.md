### 4.1 关键事实口径冲突需要强制校验

会话中多个高权重事实存在口径冲突，但 PM 没有逐条进行事实裁决。

| 争议点 | 会话中的冲突表现 | 影响 |
| --- | --- | --- |
| 扣非净利润 | 多数 Agent 使用 `-27%`，激进分析师引用 Memory 称“实际为 `-0.27%` 而非 `-27%`” | 直接影响盈利质量判断 |
| 减持窗口 | 有报告称“窗口期至 5 月底”，但同一会话又描述 6 月 4 日减持和剩余约 6900 万股额度 | 直接影响剩余抛压判断 |
| 北向持股比例 | 有报告写 `10.26% -> 3.23%`，也有报告写 `0.1026% -> 0.0323%` | 百分号口径错误会放大或缩小资金面判断 |

| 价格口径 | 同时使用 `38.70`、`38.88` | 影响估值、止损距离和浮亏计算 |

对应修复方案：在 PM 前增加“事实冲突仲裁”步骤，用于识别并裁决关键事实冲突。该步骤不替代 PM 决策，只为 PM 提供已裁决事实、未解决冲突和必须降权处理的信息。

建议位置：战略二轮结束之后、PM 节点之前。

建议涉及模块：

| 模块 | 建议职责 |
| --- | --- |
| `backend/app/ai/llm_engine/orchestrator.py` | 在 LangGraph 中增加事实仲裁节点，并把结果写入 state |
| `backend/app/ai/llm_engine/prompts/templates.py` | 增加事实仲裁 Markdown 模板，并给 PM 输入追加仲裁摘要 |
| `backend/app/ai/llm_engine/agents/base.py` | 仲裁节点可直接读取既有 Agent Markdown 报告，不强制前序 Agent 先结构化输出 |
| `backend/app/models/debate_message.py` | 可用 `reasoning` 或 `analysis.markdown` 保存仲裁 Markdown，无需新增表即可先落地 |

### 4.2 Memory 使用规则应内置到提示词

本次会话中 Memory 同时被用于支持减仓和支持持有。

| Memory 经验 | 风险/保守解释 | PM 解释 |
| --- | --- | --- |
| “大股东发布新减持公告或质押爆仓 -> 减仓至 10% 以下” | 当前减持触发，应减仓 | 当前只是已披露计划的执行，不属于“新减持公告” |
| “减持窗口关闭后股价企稳反弹” | 部分适用，但当前是新计划，仍有剩余额度 | 采纳，强化 `HOLD` |
| “37.8-38.0 元强支撑” | 支撑可能被减持利空打破 | 采纳，认为下行有限 |

PM 最后选择拒绝风险 Memory、采纳支持 `HOLD` 的 Memory，该裁决可以成立，但论证强度不足。

尤其是激进分析师引用 Memory 称“扣非净利润实际为 `-0.27%` 而非 `-27%`”，这类历史记忆如果与当前财务数据冲突，应默认降权，而不是作为事实纠错依据。

对应修复方案：在相关 Agent 和 PM 的提示词内指定 Memory 通用使用规则。Memory 的作用应通过提示词约束为“辅助经验”，而不是“当前事实源”。

提示词内建议固定写入以下规则：

| 规则 | 说明 |
| --- | --- | --- |
| 当前数据优先 | Memory 与当前上下文、数据库指标、公告或行情冲突时，必须以当前数据为准 |
| Memory 只作辅助 | Memory 可用于提醒历史模式、风险规则和用户偏好，但不能单独决定交易动作 |
| 规则必须匹配条件 | 风控类 Memory 必须判断触发条件是否完全匹配，不得把“部分相似”当作硬触发 |
| 过期事实自动降权 | 涉及旧窗口、旧价格、旧公告的 Memory 必须降权，不能替代当前事实 |
| 冲突交给仲裁节点 | 如果 Memory 与多个 Agent 的当前事实冲突，应进入事实仲裁，不由单个 Agent 自行改写事实 |
| 仅在影响决策时说明 | 最终报告只需说明对决策有实质影响的 Memory 规则，不需要逐条输出 Memory 使用记录 |

验收标准：

| 标准 | 说明 |
| --- | --- |
| 提示词包含通用规则 | Agent 和 PM 均应被明确要求“当前数据优先、Memory 辅助” |
| Memory 冲突不得覆盖当前事实 | 如 `-0.27%` 与 `-27%` 冲突时，应进入仲裁并默认当前数据优先 |
| 最终报告只说明实质影响 | 只在 Memory 显著影响决策时用自然语言说明 |

### 4.3 风险专家的“一票否决”没有制度化处理

风险专家和保守分析师都明确建议将格力电器仓位从约 19% 降至 10% 以下。PM 最终维持约 19% 仓位。

这不一定错误，但流程上存在两个问题。

| 问题 | 影响 |
| --- | --- |
| 风控建议和 PM 决策优先级不清 | 用户难以判断“一票否决”到底是否为硬约束 |
| PM 覆盖风控建议但解释不足 | 容易显得 PM 在选择性忽略风险 |

对应修复方案：风险专家继续输出 Markdown，但使用固定小节模板表达风险等级、建议动作、触发条件和 PM 覆盖要求。不要求风险专家输出结构化 JSON。

建议风险等级：

| 等级 | 含义 | PM 处理规则 |
| --- | --- | --- |
| 硬阻断 | 不建议交易或必须降仓 | PM 覆盖时必须单独说明原因，并解释置信度依据 |
| 强警告 | 需要仓位或止损调整 | PM 必须回应，不可忽略 |
| 观察项 | 需要持续跟踪 | PM 可纳入触发条件 |

风险专家建议输出为固定 Markdown 模板：

```markdown
# 风险专家结论

## 风险等级

强警告

## 建议动作

建议将仓位降至 10% 以下，或至少进入 PM 强制复核。

## 触发条件

| 条件 | 建议响应 |
| --- | --- |
| 剩余减持额度继续执行 | 减仓至 10% 或重新评估 |
| 扣非利润继续恶化 | 冻结加仓并重新评估持仓 |

## PM 覆盖要求

| 要求 | 说明 |
| --- | --- |
| 覆盖理由 | PM 必须说明为何不减仓 |
| 置信度说明 | 若核心风险未解除，PM 必须解释为何仍给出当前置信度 |
```

PM 覆盖风控建议时，最终报告必须包含：

| 字段 | 说明 |
| --- | --- |
| 风险建议原文摘要 | 风险专家建议了什么 |
| 覆盖理由 | PM 为什么不采纳 |
| 覆盖代价 | 如果风险专家正确，组合可能承受什么损失 |
| 替代控制 | 用什么触发器、止损或仓位限制替代 |
| 置信度调整 | 覆盖强风控后如何调整置信度 |

### 4.4 PM 置信度依据不足

PM 最终置信度为 `0.78`，但会话仍存在多个未裁决不确定性。

| 不确定性 | 影响 |
| --- | --- |
| 扣非净利润口径冲突 | 影响盈利质量判断 |
| 剩余 6900 万股减持是否还能继续执行 | 影响短期抛压判断 |
| 风险专家和 PM 对 Memory 规则触发解释相反 | 影响风控动作判断 |
| 北向资金、质押、减持窗口存在时效和口径问题 | 影响风险权重 |
| 液冷、碳化硅、华为供应商定位缺少业绩贡献量化 | 影响上行空间可信度 |

对应修复方案：PM 置信度由 PM 自主判断，提示词只提供参考规则和解释要求，不做硬编码扣分或固定阈值。

建议提示词规则：

| 情况 | PM 应如何处理 |
| --- | --- |
| 存在未解决高权重事实冲突 | 说明冲突为何不改变结论，或说明结论因此更谨慎 |
| 覆盖风险专家“硬阻断”或“强警告” | 说明覆盖理由、替代风控和对置信度的影响 |
| Memory 与当前数据冲突 | 说明采用当前数据还是 Memory 经验，以及原因 |
| 数据时效超过关键窗口 | 说明数据时效不足如何影响判断 |
| 仓位为组合第一大持仓且争议明显 | 说明组合暴露对置信度和执行纪律的影响 |
| 有明确风控触发器且可执行 | 可以作为提高决策可执行性的理由，但不自动提高置信度 |

以本次会话为例，PM 可以维持 `0.78` 或选择更低置信度，但必须解释扣非口径冲突、减持窗口不确定、覆盖风险强建议和 Memory 解释冲突为什么没有改变最终信心分。重点是“可解释”，不是硬性扣分。

### 4.7 组合层面的仓位比较不够充分

PM 最终维持约 19% 仓位，但没有充分比较几个可行方案。

| 方案 | 缺少的比较 |
| --- | --- |
| 维持 19% | 若扣非继续恶化或剩余减持继续执行，组合回撤是多少 |
| 降至 10% | 会损失多少上行弹性，换来多少风险降低 |
| 不动但设置更严格触发器 | 是否优于立即减仓 |
| 使用现金等待确认后再加仓 | 与当前持有相比的机会成本 |

对应修复方案：对组合最大持仓或争议较大的标的，PM 应比较至少三种方案。

建议模板：

| 方案 | 仓位 | 上行情景 | 下行情景 | 优点 | 缺点 | 适用条件 |
| --- | --- | --- | --- | --- | --- | --- |
| 维持 | 19% | 保留反弹弹性 | 承担减持和扣非恶化风险 | 不在低估值区卖出 | 回撤暴露较大 | 事实冲突可控，风险未触发硬阻断 |
| 降仓 | 10% | 上行收益降低 | 回撤下降 | 控制最大持仓风险 | 可能卖在底部 | 风控强警告未解除 |
| 等待确认 | 19%，但设置硬触发 | 兼顾弹性 | 依赖触发纪律 | 不立即交易 | 可能滞后 | 有明确可执行触发器 |

PM 最终选择某一方案时，应说明为什么它优于其他方案。

### 4.8 风控复核条件有些滞后或不够闭环

PM 最终设置了价格、财报和减持触发器，但部分条件不够闭环。

| 触发条件 | 问题 |
| --- | --- |
| 跌破 MA60 且连续 3 日无法收回 | 如果减持导致快速下跌，3 日确认可能滞后 |
| 跌破 36.80 元清仓 | 会话未生成订单，需区分策略止损与已挂止损 |
| 半年报扣非降幅大于 25% 且毛利率低于 26% | 当前扣非已为 `-27%`，但又要求毛利率同时低于 26%，门槛偏高 |
| 超出原计划的新减持公告才提前减仓 | 剩余计划内减持继续执行没有明确动作 |

对应修复方案：在风险专家和 PM 报告中把“剩余减持继续执行”“质押预警”“扣非继续恶化”“价格破位”等事项作为复核条件和执行提示写清楚，避免 PM 只依赖价格确认。

建议报告层复核条件：

| 复核条件 | 条件 | 报告层建议 |
| --- | --- | --- |
| 剩余减持继续执行 | 原计划内剩余减持再次落地 | 重新评估是否降至 10%，不再只等待新减持公告 |
| 质押风险升级 | 股价接近质押预警区或出现补充质押公告 | 在风险专家报告中标为“强警告”或“硬阻断”，PM 若覆盖需解释 |
| 盈利质量继续恶化 | 半年报扣非降幅未收窄，或毛利率继续下行 | 降仓或冻结加仓建议 |
| 价格快速破位 | 跌破关键支撑且伴随放量 | 报告中要求 PM 说明是否需要立即复核，而不是等待 3 日确认 |
| 策略止损未落单 | 仅报告止损但未生成订单 | 报告中明确提示“策略止损未自动执行” |

### 4.9 非 PM 报告的持久化置信度容易被误读

非 PM Agent 当前主要输出 Markdown 字符串，本身没有单独的输出置信度字段。但 `debate_messages` 表和 API 响应对所有消息都有 `confidence` 字段；字符串报告持久化时默认写入 `0.0`，因此历史详情中容易把“未填写”误读为“低置信”。PM 则通过 `PMDecision.confidence_score` 写入真实置信度，例如本次为 `0.78`。

对应修复方案：不要求非 PM Agent 结构化输出置信度。若系统保留消息级 `confidence` 字段，非 PM Markdown 报告未填写时应使用 `null` 或前端不展示；如 Agent 想说明信心来源，可在 Markdown 报告中用自然语言说明。

建议规则：

| 规则 | 说明 |
| --- | --- |
| 未填写置信度时使用 `null` | 避免把未填误解为低置信 |
| Markdown 中说明信心依据 | 如有必要，用自然语言说明信心来自数据一致性、时效性还是推断 |
| 置信度不参与硬决策 | 只辅助 PM 判断，不替代证据和风控等级 |

### 8.3 验证方案

| 验证项 | 方法 |
| --- | --- |
| 事实冲突仲裁 | 构造含冲突事实的假 Agent 输出，确认仲裁 Markdown 能选择当前数据并降权 Memory |
| Memory 通用规则 | 构造 Memory 与当前数据冲突的输入，确认提示词要求当前数据优先且只在显著影响结论时自然语言说明 |
| 风控覆盖 | 风险专家输出“强警告”时，确认 PM 必须回应 |
| 置信度说明 | 存在重大未解决冲突时，确认 PM 对置信度给出合理解释 |
| 仓位方案比较 | 对第一大持仓，确认 PM 输出至少三种方案比较 |
| 前端展示 | 确认事实仲裁节点能在历史详情中查看；Memory 影响和风控覆盖随各 Agent Markdown 展示即可 |

测试注意事项：提示词效果不建议通过字符串断言锁死。事实仲裁节点和风险专家可检查是否包含必要 Markdown 小节；第二轮交叉质询复用既有辩论反驳章节，长期催化剂映射使用 Markdown 表格。PM 重点检查是否覆盖事实仲裁、风险覆盖、仓位比较和置信度说明，而不是断言具体 prompt 文案。

### 8.4 风险与取舍

| 风险 | 取舍建议 |
| --- | --- |
| 流程变长，Token 成本上升 | P0 只仲裁高权重事实，不处理所有细节 |
| Markdown 模板过多导致输出僵硬 | 只固定必要小节，不限定具体措辞 |
| 仲裁节点结构化成本过高 | 仲裁节点先输出固定 Markdown，不强制 JSON schema |
| PM 变得过于保守 | 风控不是自动否决，置信度也不硬编码扣分，而是强制解释依据 |
| Memory 被过度降权 | Memory 仍可作为经验，但不能覆盖当前事实 |
| 测试难以覆盖 LLM 文案 | 测固定小节、关键字段和决策流程，不测试 prompt 具体表达 |
