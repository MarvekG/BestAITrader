# 经验记忆系统设计

本文描述经验复盘模块的下一步产品方向：在不做自动选股、不改变现有研究和模拟交易定位的前提下，把经验复盘升级为以 Memory 为核心的经验沉淀、展示和复用闭环。

## 1. 产品定位

经验复盘不应只是判断某次 AI 辩论结论是否正确，而应帮助用户持续回答三个问题：

1. 当初的 PM 判断是否被后续市场验证？
2. 当时使用的信号中，哪些被验证、哪些被证伪、哪些只是噪音？
3. 下次 debate 或 PM 决策应该具体改什么？

本设计采用 Memory-first 方案：

- 经验和证据链的主存储仍然是记忆系统。
- Best-AI-Trader 内部的经验表只作为展示索引、筛选缓存和 UI 加速，不作为经验事实来源。
- 复盘模块负责生产高质量经验，Memory 负责沉淀和召回，经验库入口负责让用户看见、检索和审计这些经验。

一句话概括：复盘模块负责把交易判断变成高质量经验，Memory 负责保存经验和证据，经验库只负责让用户看得见、找得到、用得上。

## 2. 设计原则

- 不做自动选股：系统不主动推荐用户买哪只股票。
- 不直接自动交易：复盘结果只用于研究、模拟交易和流程改进。
- Memory 是事实来源：经验正文、证据链和可复用规则应写入 Memory。
- 经验表只是目录：业务库中的经验索引只保存筛选、展示和跳转所需的轻量字段。
- 输出必须结构化：每次复盘必须产出固定的三类结论，避免 LLM 生成泛泛总结。
- 证据必须可见：用户能看到每条经验来自哪次复盘、哪些市场结果、哪些 PM 判断和哪些信号验证。
- 经验必须能反哺决策：后续 debate 和 PM 决策应能召回相关经验，并说明是否采纳。

## 3. 整体链路

```text
Debate Session
  -> 满足 5/20/60 日复盘条件
  -> 经验复盘
  -> 产出结构化复盘结论
  -> 写入 Memory：经验 + 证据链
  -> 同步轻量 Experience Index 到本系统
  -> 前端经验分析页内经验库 Tab 展示和筛选
  -> 后续 Debate / PM 从 Memory 召回经验
```

这里的关键是 `Experience Index` 不是经验本体，只是 Memory 的可视化目录。

系统中的三类数据职责如下：

| 数据 | 职责 |
| --- | --- |
| Memory | 保存经验正文、证据链、可复用规则和失效边界，是经验事实来源。 |
| ExperienceReviewEvent | 保存复盘运行事件、结果 payload、工具调用和运行状态。 |
| Experience Index | 保存轻量展示索引，用于经验库列表、筛选、统计和跳转。 |

## 4. 复盘候选自动发现

复盘页不应让用户自己从历史 session 中猜哪些可以复盘。后端应提供复盘候选扫描能力。

扫描流程：

1. 扫描当前用户所有 `completed` debate session。
2. 找到每个 session 最新 PM 决策时间。
3. 统计决策日之后目标股票可用日 K 数量。
4. 判断是否满足 `5D`、`20D`、`60D` 复盘条件。
5. 判断对应周期是否已经复盘、正在复盘或曾经失败。
6. 返回带状态的候选列表。

复盘周期：

```text
5D: 决策日后至少 6 条日 K，包括决策日和第 5 个交易日
20D: 决策日后至少 21 条日 K
60D: 决策日后至少 61 条日 K
```

候选状态建议：

```text
not_ready
ready_5d
ready_20d
ready_60d
reviewing
reviewed
failed
```

前端复盘页展示：

```text
可 5D 复盘
可 20D 复盘
可 60D 复盘
等待更多数据
已复盘
复盘失败
```

同一个 session 可以分别做三次复盘：

```text
5D 初步复盘
20D 标准复盘
60D 中期复盘
```

默认优先级建议：

```text
20D > 60D > 5D
```

原因：

- 5D 太短，更适合短线验证和早期纠错。
- 20D 最适合作为默认复盘口径。
- 60D 更适合验证中期逻辑，不需要高频自动运行。

## 5. 复盘输出三件套

每次经验复盘必须产出三类东西，并将其作为 schema 约束，而不是只写在 prompt 中。

### 5.1 原始判断是否正确

判断 PM 当时的 `buy / sell / hold / reduce / add` 是否被后续市场验证。

判断依据不应只看绝对涨跌，还应结合：

- 决策后绝对收益；
- 相对沪深 300 或其他基准收益；
- 相对行业收益；
- 最大回撤；
- PM 原始置信度和目标仓位；
- 原始交易周期和策略。

输出示例：

```json
{
  "verdict": "incorrect",
  "score": 35,
  "pm_decision": "add",
  "outcome_basis": "20D 收益 -6%，跑输沪深 300 4%，最大回撤 11%",
  "reasoning": "PM 将技术突破作为加仓依据，但突破后成交未持续放大，且行业相对收益转负。"
}
```

### 5.2 哪些信号被验证或证伪

复盘应从 debate timeline 和 PM 决策中提取当时使用过的信号，并拆成三类：

- 被验证信号；
- 被证伪信号；
- 噪音信号。

输出示例：

```json
{
  "validated_signals": [
    {
      "signal": "行业 Beta 转弱",
      "evidence": "复盘期内同行业平均收益弱于指数，目标股票同步走弱。",
      "impact": "high",
      "lesson": "行业相对强弱应在 PM 仓位决策前被显式检查。"
    }
  ],
  "invalidated_signals": [
    {
      "signal": "技术突破",
      "evidence": "突破后 3-5 个交易日未持续放量，并跌回突破平台。",
      "impact": "medium",
      "lesson": "无成交确认的突破不能作为加仓主因。"
    }
  ],
  "noise_signals": [
    {
      "signal": "短期新闻热度",
      "reason": "新闻热度未对应成交扩张，也未形成行业相对收益。"
    }
  ]
}
```

### 5.3 下次 Debate 或 PM 决策要改什么

复盘必须输出可执行的流程改进，而不是抽象观点。

改进方向包括：

- 哪个 Agent 下次必须补充什么证据；
- PM 下次如何调整仓位；
- 什么情况下不能加仓；
- 什么情况下必须降低置信度；
- 哪些信号以后只能作为辅助，不能作为主因；
- 风控 Agent 下次必须检查哪些否决条件。

输出示例：

```json
{
  "debate_changes": [
    "技术面 Agent 必须验证突破后的成交持续性，不能只报告突破形态。"
  ],
  "pm_changes": [
    "PM 输出目标仓位前必须检查行业相对强度，行业相对收益转负时不得提高仓位。"
  ],
  "risk_control_changes": [
    "若 5D 内跌破入场价 3% 且板块同步转弱，应触发减仓或重新辩论。"
  ]
}
```

## 6. Memory 写入设计

复盘完成后，系统应将经验和证据链写入 Memory。Memory 写入不是附属动作，而是经验复盘的核心产物。

每次复盘建议写入 1-3 条 Memory。每条 Memory 应尽量自包含，覆盖以下要素：

```text
经验主题
适用对象
复盘周期
原始判断
后验结果
被验证信号
被证伪信号
主导驱动
可复用规则
失效边界
证据链
```

示例：

```text
经验主题：无成交确认的技术突破不能支撑加仓
适用对象：600000.SH / 银行业 / 波段策略
复盘周期：20D
原始判断：PM 建议加仓，理由是技术突破和情绪升温
后验结果：20D 收益 -6%，跑输沪深 300 4%，最大回撤 11%
被验证信号：行业 Beta 转弱
被证伪信号：技术突破、短期情绪
主导驱动：行业相对收益转负，突破后成交未持续放大
可复用规则：如果突破后 3-5 个交易日没有成交放大，且行业相对收益转负，PM 不应提高仓位
失效边界：强政策催化或业绩超预期时需要重新评估
证据链：session_id、review_run_id、PM message、market outcome、driver review
```

写入要求：

- 不写普通流水账。
- 不写空泛背景信息。
- 不只重复“正确/错误”标签。
- 必须包含后验市场结果。
- 必须包含被验证和被证伪的信号。
- 必须包含未来可复用的规则和失效边界。
- 必须能让未来 debate 或 PM 在召回后理解为什么这条经验可信。

## 7. Experience Index 设计

业务库可以新增轻量经验展示索引，但它不是经验主存储。

Experience Index 只保存列表展示、筛选、统计和跳转所需字段，例如：

```text
memory_observation_id / memory_source_id
review_run_id
session_id
stock_code
stock_name
industry
strategy
review_horizon
outcome_label
correctness
tags
summary
importance
created_at
```

不建议在 Experience Index 中复制完整经验正文和完整证据链。完整内容应从 Memory 获取，或从复盘结果事件中回查。

职责边界：

```text
ExperienceIndex = Memory 的可视化目录
Memory = 经验和证据的事实来源
ExperienceReviewEvent = 复盘运行记录
```

这样的好处：

- Memory 仍然是系统学习和召回的核心。
- 经验库入口查询更快。
- 支持按股票、行业、策略、失败教训、仓位纪律筛选。
- 支持统计数量和跳转原始复盘。
- 不会出现业务表和 Memory 各自保存一套经验正文导致事实不一致。

## 8. 经验分析页内经验库 Tab

在“经验分析”页内新增“经验库”Tab。它不是独立知识库，也不作为独立导航页面，而是 Memory 经验的可视化入口。

页面目标：

- 查看已成功写入 Memory 并同步到 Experience Index 的历史复盘经验。
- 按股票、行业、策略、复盘周期、标签、结果、关键词筛选。
- 从经验反查原始复盘证据链。
- 让 Memory 从模型底层能力变成用户可见、可审计、可复用的资产。

筛选条件：

```text
股票
行业
策略
复盘周期：5D / 20D / 60D
原始判断正确性
结果：赚钱 / 亏钱 / 跑赢 / 跑输
失败教训
仓位纪律
信号类型
重要性
时间范围
关键词
```

列表展示当前只展示成功写入 Memory 并同步索引的经验，不单独显示失败 Memory 写入状态。失败状态在复盘结果页展示。

列表展示：

```text
经验摘要
股票 / 行业
策略
复盘周期
结果标签
原始判断是否正确
失败教训标签
仓位纪律标签
Memory source/event id
创建时间
```

详情展示：

```text
经验摘要
Memory source/event id
原始判断是否正确
信号验证/证伪
下次 Debate/PM 改进
复盘事件 payload 中的证据链
关联 review_run
关联 session
```

详情页应明确告诉用户：

```text
这条经验来自 Memory；当前经验库仅为展示索引。完整经验事实以 Memory 为准，详情页只展示索引摘要和复盘事件上下文。
```

## 9. 复盘结果页改造

复盘完成后，页面应直接展示以下区域：

```text
一、原始判断是否正确
二、信号验证/证伪
三、下次 Debate/PM 改进
四、写入 Memory 的经验
五、证据链
```

Memory 写入区必须显眼展示。每条 Memory 展示：

```text
写入状态
经验摘要
重要性
适用范围
关联股票 / 行业 / 策略
查看证据链
查看 Memory 标识和复盘上下文
```

如果 Memory 写入失败，页面不应静默失败，应显示：

```text
复盘完成，但经验写入 Memory 失败。
```

否则用户会误以为系统已经学到了经验。

## 10. 后续 Debate / PM 经验召回

经验复盘的最终价值不只是展示历史，而是反哺未来判断。

后续 Debate 或 PM 阶段，Memory 召回应优先查询与当前目标股票、当前分析上下文直接相关的历史经验。当前实现中的 Memory 工具由当前目标股票上下文驱动，不应假设已经支持跨股票的通用行业级或策略级主动检索。

```text
同一股票的历史经验
当前上下文召回到的失败教训
当前上下文召回到的仓位纪律
当前上下文召回到的信号验证/证伪案例
当前上下文召回到的市场环境适用边界
```

PM 决策前应显式输出：

```text
本次使用了哪些历史经验
哪些经验适用
哪些经验不适用
为什么
这些经验如何影响本次仓位、置信度和风险控制
```

历史经验不是必须服从的规则。PM 采纳历史经验前必须检查当前交易频率、交易策略、市场环境、证据状态是否匹配，并说明历史经验是否可能已经过时。如果历史经验提示高风险，而 PM 仍维持较高置信度或较高仓位，PM 必须解释为什么本次情况不同。

## 11. 分期建议

### 第一期：复盘候选 + 三件套 + Memory 可见

目标是让经验复盘变得可用、可见、可审计。

范围：

- 复盘页自动列出满足 5D/20D/60D 条件的 session。
- 发起复盘时选择复盘周期。
- 复盘输出强制三件套。
- 复盘完成后展示写入 Memory 的经验。
- Memory 写入失败时显式提示。

### 第二期：Experience Index + 经验分析页内经验库 Tab

目标是让历史经验能被主动检索和管理。

范围：

- 建立轻量 Experience Index。
- 复盘写入 Memory 成功后，同步展示索引。
- 在经验分析页内新增经验库 Tab。
- 支持股票、行业、策略、失败教训、仓位纪律筛选。
- 详情页展示 Memory 标识、索引摘要和复盘事件证据链；完整经验事实以 Memory 为准。

### 第三期：经验反哺 Debate / PM

目标是形成“复盘 -> 记忆 -> 决策”的闭环。

范围：

- Debate / PM 召回相关经验。
- PM 决策中明确列出经验使用情况。
- 高风险历史经验触发 PM 强制解释。
- 根据召回记忆和复盘证据归纳出的重复问题优化 prompt 和流程检查，不预设固定高频问题清单。

## 12. 产品边界

- 本设计不改变项目的研究、教育和模拟交易定位。
- 本设计不构成投资建议系统设计。
- 本设计不新增自动真实资金交易能力。
- 本设计不把经验库作为数据再分发或金融信息服务。
- 所有外部数据、行情、新闻和 LLM 服务仍需由部署方自行确认授权和合规边界。

## 13. 最终目标

经验记忆系统的最终目标是让系统从“每次重新分析”变成“每次分析都带着历史教训前进”。

理想状态：

1. 复盘模块持续生产高质量经验。
2. Memory 保存经验、证据和失效边界。
3. 经验库让用户看见和筛选历史教训。
4. 后续 Debate / PM 自动召回相关经验。
5. PM 决策显式说明经验如何影响判断。
6. 系统逐步减少重复犯错，形成可审计、可演进的投研闭环。
