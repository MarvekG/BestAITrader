# 记忆协议提示词设计

本文描述 Debate、PM 与经验复盘之间统一使用 Memory 的提示词协议。目标是让复盘写入的经验能被后续辩论稳定召回，同时保留 LLM 自主判断何时追加查询、何时追加写入的能力。

## 1. 目标

当前系统已经具备 `recall_memory` 和 `write_memory` 工具，但记忆使用主要依赖各角色 prompt 中的原则性说明。这样会导致两个问题：

- 复盘写入的经验主题不稳定，未来 Debate/PM 不一定按同一主题召回。
- 多个经验主题容易被揉在一条 Memory 里，召回时难以匹配具体问题。

本设计希望统一三类场景的协议：

- 经验复盘写入 Memory。
- Debate/PM 召回 Memory。
- Debate 内部生成新经验时写入 Memory。

## 2. 非目标

- 不新增数据库表。
- 不新增 Memory 工具参数。
- 不在代码中硬编码强制主题检查。
- 不依赖关键词匹配来判断 Memory 是否合格。
- 不把 Experience Index 变成经验事实来源。
- 不把所有固定主题合并写成一条超长 Memory。

## 3. 核心原则

### 3.1 固定主题通过提示词约束

固定主题只写入系统提示词和工具说明，不通过代码逻辑强制执行。

原因：

- 记忆系统需要保持泛化能力，不能变成关键词规则系统。
- 不同股票、策略、频率和市场环境下，具体内容必须由 LLM 基于证据归纳。
- 代码只提供工具和上下文，判断哪些主题有新增价值仍由 LLM 执行。

### 3.2 固定主题，不固定答案

协议只固定“应覆盖哪些经验主题”，不预设固定高频问题清单。

例如：

- 可以固定要求写入“风险控制经验”。
- 不应固定要求每次都检查“无成交确认追涨”或“高股息陷阱”。

具体风险、误判、触发条件和失效边界必须从本次证据、召回记忆和市场结果中归纳。

### 3.3 不同主题分次写入

每个 `write_memory` 调用应只承载一个主主题。

如果一次复盘同时产出以下三类经验：

- 主导驱动验证。
- 风险控制纪律。
- Debate/PM 流程改进。

则应分成三次 `write_memory`，而不是揉成一条 Memory。

原因：

- 召回 query 往往围绕单个问题，单主题 Memory 更容易命中。
- 单主题 Memory 更便于审计和复用。
- 多主题长文本会降低未来 LLM 判断“哪些经验适用、哪些经验不适用”的清晰度。

### 3.4 主题化召回由 LLM 自主决定

PM 和有记忆权限的 Debate Agent 不按角色硬性限制召回主题，也不要求每轮机械召回固定主题。LLM 应根据当前不确定性、当前证据缺口、交易频率、交易策略和市场环境，自主决定是否召回以及召回哪些主题。

主题化 recall 的作用是让召回 query 更聚焦，便于复盘经验回流；自主 recall 的作用是避免低相关记忆污染上下文，并覆盖协议主题之外的新问题。

### 3.5 记忆权限边界

事实优先的一阶分析师不应使用 Memory，避免历史经验污染当前事实核验、实时情绪、新闻事件或政策原文判断。禁用记忆的角色包括：

- `FUNDAMENTAL` / 基本面分析师。
- `TECHNICAL` / 技术面分析师。
- `CAPITAL_FLOW` / 资金流分析师。
- `SENTIMENT` / 情绪分析师。
- `NEWS_ANALYST` / 新闻分析师。
- `POLICY_ANALYST` / 政策分析师。

允许使用记忆的角色包括：

- `RISK_CONTROL` / 风控分析师。
- `BULL` / 多头研究员。
- `BEAR` / 空头研究员。
- `AGGRESSIVE` / 激进分析师。
- `CONSERVATIVE` / 保守分析师。
- `NEUTRAL` / 中性分析师。
- `PORTFOLIO_MANAGER` / 投资经理。

公共 prompt 只描述“若当前角色可使用记忆工具”时应遵循的统一协议，不在公共 prompt 中列出禁用角色。禁用角色必须在各自角色 prompt 中明确禁止 `recall_memory` 和 `write_memory`，运行时工具绑定也必须与该白名单保持一致。

### 3.6 固定写入与自主写入并存

复盘和 Debate 内部写入 Memory 时，应优先按固定主题拆分写入；同时允许 LLM 对新出现的高价值模式追加写入。

自主写入必须满足：

- 有新增可复用价值。
- 包含触发条件、证据、动作和失效边界。
- 不只是复述当前事实或普通背景。

## 4. 统一主题

第一阶段固定 5 个写入/召回主题。主题名用于提示词中的协议标识，不要求后端解析。

`decision_outcome` 仍然有必要保留。它用于复盘阶段记录“原始决策和后验结果”的对应关系，是判断 PM 结论是否正确、同类策略历史结果如何的重要事实来源。但它不应成为固定召回要求；只有当本轮需要对比历史类似决策结果，或复盘时存在明确后验结果时，才写入或召回该主题。

### 4.1 `decision_outcome`

用途：记录原始决策和后验结果。

适合写入：

- 原 PM 动作、目标仓位、置信度、止损/加仓计划。
- 决策后的收益、回撤、相对收益。
- 原始判断是正确、部分正确、错误还是无法判断。
- 哪些结论经得起后验检验，哪些存在偏差。

适合召回：

- PM 裁决前检查历史类似决策结果。
- 复盘时对比过去同类策略是否有效。

### 4.2 `driver_validation`

用途：记录信号和驱动的验证/证伪关系。

适合写入：

- 被市场验证的信号。
- 被市场证伪的信号。
- 看似合理但对实际涨跌贡献小的噪音信号。
- 主导涨跌因素和被排除的伪因。

适合召回：

- Bull/Bear/Neutral 争论某个驱动是否重要时。
- PM 判断当前核心驱动是否已有历史失效案例时。

### 4.3 `risk_control`

用途：记录仓位、止损、加仓、退出和失效条件。

适合写入：

- 首次仓位是否过大或过小。
- 加仓条件是否过早或过宽。
- 止损条件是否缺失或失效。
- 回撤、流动性、板块 Beta、事件落地等风险边界。

适合召回：

- Risk Control Agent 输出风控建议前。
- PM 输出目标仓位、止损和加仓计划前。

### 4.4 `strategy_fit`

用途：记录经验适用的交易频率、交易策略和市场环境。

适合写入：

- 该经验适用于日内、波段、中长线还是长期。
- 该经验适用于价值、趋势、动量、事件驱动还是其他策略。
- 该经验在什么市场环境下适用或失效。
- 历史经验是否过时，以及为什么可能过时。

适合召回：

- PM 判断历史经验能否迁移到当前交易策略时。
- Debate Agent 引用历史经验前做适用性校验时。

### 4.5 `process_improvement`

用途：记录 Debate、PM 和 Risk 流程下次应改什么。

适合写入：

- 哪个 Agent 下次必须补充什么证据。
- 哪类推理错误需要避免。
- PM 下次如何调整仓位、置信度或退出设计。
- Risk Control 下次必须检查哪些否决条件。

适合召回：

- 新一轮 Debate 开始时，用于提醒 Agent 避免历史流程缺陷。
- PM 裁决前，用于检查本轮是否再次犯同类流程错误。

## 5. 写入协议

### 5.1 复盘写入

经验复盘完成时，LLM 应按主题拆分写入 Memory。

推荐顺序：

1. `decision_outcome`：如果原始 PM 结论有明确后验结果。
2. `driver_validation`：如果能区分被验证、被证伪和噪音信号。
3. `risk_control`：如果仓位、止损、加仓、退出或回撤管理有教训。
4. `strategy_fit`：如果经验的适用频率、策略或市场环境存在明显边界。
5. `process_improvement`：如果能提炼出未来 Debate/PM/Risk 的流程检查项。

不是每次复盘都必须写满 5 条。如果某主题没有新增经验，应跳过。

### 5.2 Debate 内部写入

有记忆权限的 Debate Agent 和 PM 在本轮形成新增可复用经验时，也应按主题拆分写入。

Debate 内部写入通常不具备后验结果，因此应避免伪造 `decision_outcome` 的后验收益。更适合写入：

- `risk_control`：本轮明确形成新的风控规则或失效条件。
- `strategy_fit`：发现某历史经验不适配当前交易频率或策略。
- `process_improvement`：发现流程层面需要未来持续检查的问题。

如果只是当前事实判断，不应写入 Memory。

### 5.3 单条 Memory 内容格式

单条 Memory 建议使用以下自然语言结构。格式由 prompt 约束，不需要后端解析。

```text
[MEMORY_TOPIC: risk_control]
对象: 交通银行(601328.SH)
交易频率: 中长线持有
交易策略: 价值投资
场景: 中长线持有 / 价值投资 / 银行板块 Beta 为负
经验: 左侧建仓时首次仓位不应超过 15%，需预留加仓弹药。
触发条件: 板块月跌幅扩大、ETF 赎回或风格切换导致个股被系统性压制。
未来动作: PM 输出仓位前先检查板块资金流和行业相对收益；Risk Control 给出最大仓位和硬止损。
失效边界: 板块资金净流入逆转、市场风格从科技回归均衡、或个股出现超预期基本面催化。
证据: 复盘期收益 -1.65%，最大回撤 -2.38%，板块 5 月 -2.75%。
```

要求：

- 一条 Memory 只写一个主主题。
- 必须同时写清真实股票名和股票代码。
- 必须写清交易频率和策略，若无法确认则说明缺失。
- 必须写清触发条件、未来动作和失效边界。
- 复盘写入必须包含后验市场结果或信号验证证据。
- Debate 内部写入不能伪造未来后验结果。

## 6. 召回协议

### 6.1 自主 recall 选择

有记忆权限的 Agent 和 PM 应根据当前任务自主决定是否调用 `recall_memory`。召回主题不按 Agent 角色固定，也不要求每轮检查所有主题。

可选择的召回主题包括：

- `decision_outcome`：当需要对比历史类似 PM 决策结果、后验收益、回撤或结论正确性时召回。
- `driver_validation`：当需要判断当前核心驱动、信号或噪音是否已有历史验证/证伪经验时召回。
- `risk_control`：当需要输出仓位、止损、加仓、减仓、退出或失效条件时召回。
- `strategy_fit`：当需要判断历史经验是否适配当前交易频率、交易策略或市场环境时召回。
- `process_improvement`：当需要检查本轮 Debate、PM 或风控流程是否可能重复历史流程缺陷时召回。

如果当前证据已经足够，LLM 可以不召回记忆；如果召回某个主题但最终不采用，应在报告中说明原因。

### 6.2 召回边界

不要求所有 Agent 都召回所有主题。召回应服务于当前最重要的不确定性，而不是为了覆盖协议主题而扩大工具调用。

适合召回的情况：

- 当前结论依赖某个历史上经常误判的驱动或信号。
- 当前交易计划需要明确仓位、止损、加仓或退出纪律。
- 当前策略、频率或市场环境与历史经验可能不匹配。
- 当前流程可能重复历史 Debate、PM 或风控缺陷。
- 当前 PM 决策需要对比历史类似决策的后验结果。

不适合召回的情况：

- 当前事实证据已经足够支撑低风险结论。
- 召回主题与当前核心不确定性无关。
- 只是为了形式完整而机械调用记忆工具。

### 6.3 Query 写法

Query 应同时包含真实股票名和股票代码、主题、策略频率和 2-5 个当前关键变量。

示例：

```text
交通银行(601328.SH) [MEMORY_TOPIC: risk_control] 中长线 价值投资 银行Beta 仓位 止损 加仓
交通银行(601328.SH) [MEMORY_TOPIC: driver_validation] 业绩说明会 高股息 PB低估 板块资金流
交通银行(601328.SH) [MEMORY_TOPIC: process_improvement] Debate PM 风控 检查项 催化验证
```

不要写：

```text
查一下历史经验
当前股票以前怎么样
有没有类似案例
```

## 7. 采纳协议

召回记忆后，最终报告必须说明：

- 采用了哪些 Memory 经验。
- 每条经验如何影响本轮判断、仓位、止损或置信度。
- 哪些 Memory 没有采用。
- 没采用的原因：交易频率不匹配、策略不同、市场环境变化、证据状态不同、经验过时或当前事实不支持。

这部分仍通过提示词要求，不做代码硬校验。

## 8. 与 Experience Index 的关系

Experience Index 继续只做展示、筛选、统计和跳转。

第一阶段不要求 Experience Index 解析 `[MEMORY_TOPIC: ...]`。如果未来需要更强筛选，可以在不改变 Memory 事实来源的前提下，把主题解析为索引字段。

当前阶段：

- Memory 正文是事实来源。
- `[MEMORY_TOPIC: ...]` 是自然语言协议的一部分。
- Experience Index 可原样展示 Memory 摘要。

## 9. 提示词落点

第一阶段提示词落点如下：

- `backend/app/ai/llm_engine/prompts/templates.py`
  - 公共 Memory 边界、统一主题、召回协议、写入协议和采纳协议。
  - 有记忆权限角色不重复描述 `recall_memory` / `write_memory` 细则，统一继承公共协议。
  - PM prompt 只保留 PM 决策职责和 `previous_pm_decision` 对比要求，不重复公共记忆协议。
  - 禁用记忆角色在各自角色 prompt 中明确禁止 `recall_memory` 和 `write_memory`。
- `backend/app/ai/llm_engine/roles.py`
  - `MEMORY_ENABLED_AGENT_NAMES` 只保留可使用记忆的角色。
- `backend/app/ai/experience/workflow.py`
  - 复盘写入 Memory 时按主题拆分。
  - 不同主题分次调用 `write_memory`。
  - 复盘输出仍保留三件套 schema。
- `backend/app/ai/agentic/memory_tools.py`
  - 工具 docstring 补充主题协议说明。

不建议第一阶段新增：

- 新数据库表。
- 新配置项。
- 新工具参数。
- 代码级主题校验器。

## 10. 测试策略

第一阶段测试只验证 prompt 协议被注入，不验证 LLM 一定执行。

建议测试：

- 公共 agent prompt 包含固定主题列表。
- 公共 agent prompt 包含自主 recall 选择和采纳/不采纳说明。
- 有记忆权限角色 prompt 不重复公共 recall/write 规则。
- 禁用记忆角色 prompt 明确禁止 `recall_memory` 和 `write_memory`。
- 运行时只给允许使用记忆的角色绑定 Memory 工具。
- 复盘 prompt 要求按主题分次 `write_memory`。
- 复盘 prompt 明确禁止把多个主题揉成一条 Memory。
- Memory 工具描述包含 `[MEMORY_TOPIC: ...]` 协议示例。
- 测试样例不得复用线上或复盘真实样本，避免污染记忆系统泛化。

## 11. 审核问题

需要确认以下点后再实施：

1. 第一阶段固定 5 个主题是否合适：`decision_outcome`、`driver_validation`、`risk_control`、`strategy_fit`、`process_improvement`。
2. PM 和 Debate Agent 不限制固定召回主题，由 LLM 根据当前不确定性自主决定是否召回以及召回哪些主题。
3. `decision_outcome` 保留为有明确后验结果时的写入主题，以及需要对比历史类似决策结果时的可选召回主题。
4. 单条 Memory 是否必须带 `[MEMORY_TOPIC: ...]` 标记。
5. 是否接受第一阶段只做提示词、工具白名单和测试，不做代码级主题硬校验。
6. 是否接受 `FUNDAMENTAL`、`TECHNICAL`、`CAPITAL_FLOW`、`SENTIMENT`、`NEWS_ANALYST`、`POLICY_ANALYST` 禁用记忆，其他二阶辩论、风控和 PM 角色允许使用记忆。
